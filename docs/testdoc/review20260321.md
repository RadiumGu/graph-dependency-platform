## 【AWS PSA 视角】

### 整体判断

这是一个**完成度很高的混沌工程自动化平台**，架构清晰，FIS + Chaos Mesh 双后端设计符合企业混沌工程的真实需求。5 Phase 执行流程、Neptune 图谱驱动的目标解析、AI 假设生成+优先级排序+闭环学习的完整链路，在我审过的混沌工程方案里属于上乘水平。

### 亮点

- **Target Resolver 三级解析链**（缓存 → Neptune → AWS API 兜底）设计扎实，审计留底做得好
- **Neptune 图谱驱动**贯穿全链：目标解析、FMEA、假设生成、图谱反馈闭环，不是摆设
- **fault_injector.py 抽象层**设计合理，FaultInjector ABC + 工厂方法，后端切换干净
- **Stop Condition 双层安全网**：代码层实时检测 + FIS 原生 CloudWatch Alarm，defense-in-depth
- **Observability**：CloudWatch 自定义指标 + 结构化日志（structlog JSON），生产级

### ⚠️ 严重问题

**1. runner.py 中存在两套 Chaos Mesh 客户端实现冲突**
`runner.py` 文件内有一个完整的 `ChaosMeshClient` 类（~150 行），但 `__init__` 中实际使用的是 `from .chaos_mcp import ChaosMCPClient`。文件内的 `ChaosMeshClient` 是死代码——它直接 `import server as cm_server` 调用 Chaosmesh-MCP 的 server.py 函数，与 `chaos_mcp.py` 的 kubectl apply 方式完全不同。
→ **删除 runner.py 中的死代码 `ChaosMeshClient` 类**，避免维护者混淆。

**2. ClickHouse 硬编码 IP **  
`metrics.py` 硬编码了 `CH_HOST = "11.0.2.30"`, `CH_PORT = 8123`，且 HTTP 请求无任何认证。
→ 移到 `config.py`，加环境变量覆盖。

把DeepFlow ClickHouse 在生产环境通常需要认证，但目前没有，写到readme的todo里面，现在不修

**3. Neptune SSL 验证全局禁用（这个先忽略）**
`target_resolver.py`、`gen_template.py`、`fmea.py`、`hypothesis_agent.py` 四个文件都各自创建了 `ssl.CERT_NONE` 的 SSL context。
→ 正确做法：Neptune 使用 rds-combined-ca-bundle.pem（Neptune 共用 RDS CA），或至少通过环境变量控制是否禁用验证。

**4. FIS Stop Condition 中的变量占位符未解析**
`fis-aurora-failover.yaml` 中 `cloudwatch_alarm_arn` 使用了 `${REGION}:${ACCOUNT_ID}` 模板变量，但代码中没有模板解析逻辑。这些 ARN 会原样传给 FIS API 导致请求失败。
→ 在 `load_experiment()` 或 `fis_backend.py` 中加模板变量替换。

**5. graph_feedback.py 依赖本地代理但无 fallback**
`NEPTUNE_PROXY = "http://localhost:9876"` 硬编码，如果 neptune-proxy.py 没起，所有图谱反馈静默失败。而 target_resolver.py 是直连 Neptune 的。
→ 统一 Neptune 访问方式，或至少在 graph_feedback 中加 connectivity check。

### ⚠️ 中等问题

**6. SQL 注入风险 （这个先忽略）**
`metrics.py` 的 ClickHouse 查询直接拼接 `service` 变量到 SQL 字符串（`request_domain LIKE '%{service}%'`）。虽然内部使用，但如果 YAML 的 service 字段被篡改，可以注入 ClickHouse SQL。
→ 使用参数化查询或至少做输入校验。

**7. Neptune 查询同样有注入风险 （这个先忽略）**
`target_resolver.py` 和 `hypothesis_agent.py` 中的 OpenCypher/Gremlin 查询都是字符串拼接（`WHERE m.name CONTAINS '{service_name}'`）。
→ 使用参数化查询：OpenCypher 支持 `$param` 语法，Gremlin 支持 `bindings`。

**8. fmea.py 重复定义 Neptune 客户端**
`fmea.py` 重新定义了 `NEPTUNE_HOST`、`REGION` 等常量和 `_neptune_query` 函数，而不是复用 `config.py` + `target_resolver.py`。
→ 提取为 `runner/neptune_client.py` 共用模块。

**9. Bedrock 模型调用缺乏重试和限流处理**
`hypothesis_agent.py`、`report.py`、`learning_agent.py` 调用 Bedrock 都没有重试逻辑。Bedrock `ThrottlingException` 在批量使用时很常见。
→ 加 exponential backoff 重试，或用 boto3 的 built-in retry config。

**10. orchestrator.py 并行执行共享同一 K8s 命名空间**
`_run_parallel` 允许多实验并行跑在 `default` namespace，但 `preflight_check` 检查有无残留实验——如果并行执行，第二个实验的 preflight 会检测到第一个的实验导致失败。
→ 使用不同 namespace 隔离。

### 小问题（快速修复）

**11.** `SERVICE_TO_K8S_LABEL` 映射在 `target_resolver.py` 和 `chaos_mcp.py` 各维护了一份 → 提取为单一数据源

**12.** `runner.py` 的 `_phase4_recover` 中 FIS 后端的 `check_pods` 用的是 `self.injector.check_pods()`——`self.injector` 是 `ChaosMCPClient`，对 FIS 实验检查 Pod 健康是对的（FIS 实验影响的 Pod 在 K8s 里），但语义上容易误解

**13.** `experiment_id` 是计算属性（`@property`），每次访问都重新生成时间戳前缀，如果 `start_time` 在 Phase 0 后被修改（实际不会），ID 会变。问题不大但可以改为 `__post_init__` 固定

**14.** `resolve_targets.py` CLI 工具的 `--backend` 参数没有在 README 中显示默认值

**15.** FMEA `_calc_occurrence` 注释说主源应为 DeepFlow 自然错误率，但实际用的是 DynamoDB 实验失败率 → 实现和设计意图不一致，至少在代码注释中明确标注这是临时方案

## 【客户 PA 视角】

### 整体判断

作为架构决策者，这个平台**解决了混沌工程落地的核心痛点**——从"想做但不知怎么做"到"一键执行+自动分析"。AI 假设生成 + FMEA + 闭环学习的组合有实际价值。但有几个落地层面的关切：

### 可操作性

**16. DeepFlow 是硬依赖但 README 没有版本要求和部署指南**
整个平台的指标采集依赖 DeepFlow ClickHouse，但 README 环境依赖表只写了"DeepFlow — 可观测性平台"，没有版本要求、部署方式、ClickHouse 端口说明。对于不了解 DeepFlow 的工程师，这是个黑盒。
→ **README 必须加 DeepFlow 先决条件**：版本、ClickHouse 端口、必需的 flow_log 表 schema

**17. Chaos Mesh 部署前置条件缺失**
README 说 `>= 2.6`，但没说怎么部署。`Chaosmesh-MCP/setup-uvx.sh` 有安装脚本但没有在主 README 中引用。
→ 在主 README 加 Chaos Mesh 快速部署步骤或链接到 Chaosmesh-MCP 的 setup

**18. Neptune 图谱初始化文档缺失**
整个平台假设 Neptune 已有服务拓扑数据（Microservice 节点、Calls/DependsOn 边），但没有说明图谱是怎么初始化的。
→ 补文档：Neptune 图谱 schema + 初始化脚本/ETL 说明

### 运维负担

**19. 五个 Python 文件各自维护 Neptune 连接代码**
target_resolver, gen_template, fmea, hypothesis_agent, graph_feedback — 五处 Neptune 访问实现，三种协议（OpenCypher, Gremlin HTTP, neptune-proxy）。
→ 统一为一个 `neptune_client.py` 模块，减少维护面

**20. LLM 成本无控制 （这个忽略）**
`hypothesis_agent` 一次 generate+prioritize = 2 次 Bedrock 调用（最多 8192+4096 tokens）；`learning_agent` 又加 1 次；`report.py` 每次实验报告 1 次。`auto` 命令跑 20 个假设 = 2 次 LLM + 20 次报告 LLM = 22 次调用。
→ 加个成本估算或 `--no-llm` 开关让用户控制

**21. DynamoDB 表只有 90 天 TTL**
`report.py` 设 `ttl = int(time.time()) + 90 * 86400`。闭环学习默认也查 90 天。如果某天 TTL 清理了数据但学习 Agent 在 91 天后跑，数据就丢了。
→ TTL 至少 365 天，或可配置

### Vendor Lock-in

**22. Chaos Mesh MCP Server 是否必要？**
`Chaosmesh-MCP/` 是一个完整的 MCP Server（FastMCP），但实际执行路径 `chaos_mcp.py` 完全没有通过 MCP 协议调用它——直接 kubectl apply YAML。MCP Server 似乎只是给 Amazon Q / AI Agent 用的，对主执行引擎是多余的。
→ README 应明确 Chaosmesh-MCP 的定位（供 AI Agent 探索用，非执行路径）

## 【补充说明】实验脚本生成机制、MCP Server 定位与脚本验证

### 1. 实验脚本是怎么生成的

两条路径，取决于 backend：

**Chaos Mesh 实验（YAML 模板）：**
- `gen_template.py` 是交互式生成器。它从 **Neptune 图谱**读取服务上下文（tier、调用关系），然后根据 tier 级别自动推算稳态阈值、Stop Condition 阈值、容错系数等，生成完整的实验 YAML
- 生成逻辑是**纯规则驱动**的：`TIER_CONFIG` 定义了每个 tier 的 SR/P99 阈值，`FAULT_TOLERANCE` 定义了不同故障类别的容忍度，所有数值都是确定性计算
- 生成的 YAML 保存在 `experiments/tier0|tier1|tier2/` 目录下

**FIS 实验（YAML 模板）：**
- 同样是 YAML 格式，但 `fault.type` 以 `fis_` 前缀开头
- FIS 实验模板在 `experiments/fis/` 目录下，是**手工编写**的（非 gen_template 生成）
- 运行时由 `fis_backend.py` 把 YAML 配置转为 FIS API 调用（`create_experiment_template` + `start_experiment`）

### 2. Chaos Mesh MCP Server 在脚本生成中有没有用到

**没有。**

- `gen_template.py` 完全不依赖 Chaosmesh-MCP，它只读 Neptune 图谱 + 规则计算
- **但在执行阶段，情况比较混乱**——`runner.py` 里有**两套** Chaos Mesh 客户端：
  - **死代码 `ChaosMeshClient`**（runner.py 第 67~200 行）：直接 `import server as cm_server` 调用 Chaosmesh-MCP 的 `server.py` 函数（`cm_server.pod_kill()` 等）
  - **实际使用的 `ChaosMCPClient`**（`chaos_mcp.py`）：通过 `kubectl apply` YAML 直接操作 Chaos Mesh CRD，**完全不经过 MCP 协议**
- Chaosmesh-MCP 的 `server.py` 函数在死代码路径里被引用了，但**实际执行路径不走 MCP Server**
- MCP Server 的真实定位是给 Amazon Q / AI Agent 做探索式交互用的，不是主执行引擎的一部分

### 3. 脚本准确性验证：当前状态与差距 ⚠️

**当前已有的验证机制：**

| 验证点 | 状态 | 说明 |
| --- | --- | --- |
| YAML 语法解析 | ✅ 有 | `experiment.py` 的 `load_experiment()` 做了 YAML 解析 + 数据类映射 |
| threshold 表达式解析 | ✅ 有 | `parse_threshold()` 验证 `>= 99%` 这类表达式语法 |
| duration 格式校验 | ✅ 有 | `parse_duration()` 验证 `2m`/`30s` 格式 |
| FIS ARN 模板变量替换 | ✅ 有 | `_expand_arn()` 替换 `${REGION}/${ACCOUNT_ID}` |
| Pod 存在性检查 | ✅ 有 | Phase 0 preflight 验证目标 Pod 存在且 Ready |
| dry-run 模式 | ✅ 有 | `--dry-run` 走完全流程但跳过实际注入 |

**当前缺失的验证层：**

| 验证点 | 状态 | 风险说明 |
| --- | --- | --- |
| YAML schema 校验 | ❌ 缺失 | 没有 JSON Schema / 必填字段校验 |
| fault_type 合法性 | ❌ 缺失 | `gen_template.py` 有 `FAULT_TYPES` 白名单，但 `load_experiment()` 不校验 |
| Chaos Mesh CRD 预校验 | ❌ 缺失 | 不检查生成的 CRD YAML 是否符合 Chaos Mesh API spec |
| FIS action 参数校验 | ❌ 缺失 | 不验证 extra_params 是否包含必需字段（如 function_arn） |
| 生成后自动 dry-run | ❌ 缺失 | `gen_template.py` 生成完就存文件，不自动跑 `--dry-run` 验证 |

**核心风险**：`gen_template.py` 生成的 YAML 如果有逻辑错误（比如 threshold 设置不合理、fault_type 拼写错误），只有到运行时 Phase 2 实际注入才会暴露。没有"生成即验证"的闭环。

**建议补充的验证层：**

1. **`validate_experiment(path)` 函数**：在 `load_experiment()` 后加一层 semantic validation——检查 fault_type 是否在支持列表、必填参数是否齐全、threshold 数值是否合理（比如 stop SR 不应高于 before SR）
2. **`gen_template.py --validate`**：生成后自动调用 `load_experiment()` + `validate_experiment()` 做回归检查
3. **Chaos Mesh CRD dry-run**：`kubectl apply --dry-run=server` 验证 CRD 是否合法
4. **FIS 参数预检**：对 FIS 实验提前调用 `validate_experiment_template` 或至少检查 ARN 格式
